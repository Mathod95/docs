> Aperçu des types de templates Argo Workflows, expliquant les définitions et les invokeurs comme container, script, resource, HTTP, plugin, steps et DAG.

Dans cette leçon/article, nous allons apprendre les types de templates dans Argo Workflows.

Les templates peuvent être considérés comme des fonctions : ils définissent les instructions à exécuter. La spécification d’un workflow inclut un champ `entrypoint` qui indique le template « principal » — le template exécuté en premier.

Il existe neuf types de templates au total, répartis en deux catégories : définitions de templates et invokeurs de templates. Ci-dessous un bref aperçu de chacun.

---

### Template Container 

* Le template container est le type de template le plus courant. Il programme un conteneur Kubernetes et sa spécification suit les mêmes champs qu’un conteneur dans un Pod Kubernetes (image, command, args, env, resources, volume mounts, etc.). Utilisez-le chaque fois que vous souhaitez exécuter directement une image de conteneur.

---

### Template Script 

* Le template script est un wrapper pratique autour d’un container. Il fournit les mêmes champs que le container (image, command) et ajoute un champ `source` où vous pouvez insérer un script directement. Argo écrit le code source dans un fichier à l’intérieur du conteneur et l’exécute avec la commande fournie.

💡 **Astuce** :
Lorsque vous utilisez un template script, assurez-vous que la commande que vous fournissez lance l’interpréteur (par exemple `python`) et que l’image contient cet interpréteur. Le contenu du script est placé dans un fichier et exécuté dans le conteneur.

**Exemple de template script :**

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Workflow
metadata:
  generateName: random-number-
spec:
  entrypoint: random-number
  templates:
  - name: random-number
    script:
      image: python:3.6-alpine
      command: [python]
      source: |
        import random
        i = random.randint(100, 2000)
        print(i)
```

---

### Template Resource 

* Les templates resource permettent de créer, appliquer, supprimer ou patcher des ressources Kubernetes depuis un workflow Argo. Le manifeste à gérer est fourni dans le champ `manifest`.

**Exemple de création d’un ConfigMap :**

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Workflow
metadata:
  generateName: k8s-resource-
spec:
  entrypoint: k8s-resource
  templates:
  - name: k8s-resource
    resource:
      action: create
      manifest: |
        apiVersion: v1
        kind: ConfigMap
        metadata:
          generateName: dev-env-
        data:
          key: value
```

---

### Template Suspend 

* Un template suspend met en pause l’exécution du workflow. Vous pouvez suspendre pour une durée (champ `duration`) ou indéfiniment jusqu’à une reprise manuelle. La reprise peut se faire via CLI, API ou UI.

**Exemple de template suspend et commande de reprise :**

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Workflow
metadata:
  generateName: delay-
spec:
  entrypoint: delay
  templates:
  - name: delay
    suspend:
      duration: "20s"
```

```bash
argo -n argo resume delay-xyzb
```

### Template ContainerSet 

* Le template containerSet exécute plusieurs conteneurs dans le même pod (utile pour les sidecars ou conteneurs co-localisés partageant volumes et réseau localhost). Le champ est `containerSet` et contient un tableau de conteneurs.

**Exemple :**

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Workflow
metadata:
  generateName: container-set-template-
spec:
  entrypoint: main
  templates:
  - name: main
    containerSet:
      containers:
      - name: a
        image: rancher/cowsay
        command: [cowsay]
        args: ["Container A!!!!"]
      - name: b
        image: rancher/cowsay
        command: [cowsay]
        args: ["Container B!!!!"]
```

---

### Template HTTP 

* Un template HTTP effectue des requêtes HTTP(S). Le corps de la réponse est automatiquement exporté dans le paramètre de sortie `result`. Vous pouvez spécifier l’URL, la méthode (GET par défaut), les headers et le corps.

**Exemple :**

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Workflow
metadata:
  generateName: http-template-example-
spec:
  entrypoint: fetch-todo-item
  templates:
  - name: fetch-todo-item
    http:
      url: "https://x.y.com/todos/1"
      method: "GET"
      headers:
      - name: "Content-Type"
        value: "application/json"
```

---

### Template Plugin 

* Les templates plugin permettent aux workflows Argo d’utiliser des plugins d’exécution, étendant le comportement sans modifier le cœur d’Argo. Les plugins intégrés et tiers sont pris en charge. Dans l’exemple ci-dessous, le plugin ArgoCD est utilisé pour synchroniser des applications, donc aucun conteneur n’est exécuté ; Argo se connecte à un serveur ArgoCD pour effectuer l’action.

**Exemple avec le plugin ArgoCD :**

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Workflow
metadata:
  generateName: argocd-example-
spec:
  entrypoint: main
  templates:
  - name: main
    plugin:
      argocd:
        serverUrl: https://my-argocd.com/
        actions:
        - sync:
            project: highway-animation
            apps:
            - highway-animation
            - health-check-app
```

### Invokeurs de template

* Les templates `Steps` et `DAG` sont les deux types d’invokeurs. Ils n’exécutent pas eux-mêmes de tâches, mais définissent comment d’autres templates (définitions) sont appelés et composés dans un workflow. Utilisez `Steps` pour des flux séquentiels/parallèles et `DAG` pour des graphes de dépendances.

Avec ces types de templates, vous pouvez exprimer une large variété de patterns d’orchestration — depuis des tâches conteneur simples jusqu’à l’automatisation multi-ressources complexe et l’intégration avec d’autres systèmes.

## Types de modèles

Un modèle peut être un conteneur, un script, un DAG (graphe orienté acyclique) ou d’autres types selon la structure du workflow. Il est divisé en deux groupes : les définitions de modèles (qui définissent le travail à effectuer) et les invocateurs de modèles (qui invoquent/appellent d’autres modèles et contrôlent l’exécution).

Sélectionnez « Développer » pour vous familiariser avec les différents types de modèles.

---

### Définitions de modèles

**Conteneur (Close Container)**
Un conteneur est le type de modèle le plus courant et représente une étape du workflow qui exécute un conteneur. Il convient pour l’exécution d’applications conteneurisées ou de scripts.

**Exemple :**

```yaml
- name: whalesay
  container:
    image: docker/whalesay
    command: [cowsay]
    args: ["hello world"]
```

---

**Ressource (Close Resource)**
Une ressource représente un modèle pour créer, modifier ou supprimer des ressources Kubernetes. Elle est utile pour effectuer des opérations sur les objets Kubernetes dans le cluster sur lequel Workflows s’exécute actuellement.

**Exemple :**

```yaml
- name: k8s-owner-reference
  resource:
    action: create
    manifest: |
      apiVersion: v1
      kind: ConfigMap
      metadata:
        generateName: owned-eg-
      data:
        some: value
```

---

**Script (Close Script)**
Un script est similaire au modèle de conteneur mais permet de spécifier le script en ligne sans référence à une image de conteneur externe. Il peut être utilisé pour des scripts simples ou des commandes en une seule ligne.

**Exemple :**

```yaml
- name: gen-random-int
  script:
    image: python:alpine3.6
    command: [python]
    source: |
      import random
      i = random.randint(1, 100)
      print(i)
```

---

**Suspendre (Close Suspend)**
Un modèle de type « Suspend » suspend l’exécution, soit pour une durée donnée, soit jusqu’à reprise manuelle. La reprise peut se faire via l’interface CLI, l’API ou l’interface utilisateur.

**Exemple :**

```yaml
- name: delay
  suspend:
    duration: "20s"
```

---

### Invocateurs de modèles

**DAG (Graphe Orienté Acyclique – Close DAG)**
Un DAG permet de définir les tâches comme un graphe de dépendances. Il est utile pour les workflows ayant des dépendances complexes et une exécution conditionnelle.

**Exemple :**

```yaml
- name: diamond
  dag:
    tasks:
    - name: A
      template: echo
    - name: B
      dependencies: [A]
      template: echo
    - name: C
      dependencies: [A]
      template: echo
    - name: D
      dependencies: [B, C]
      template: echo
```

---

**Étapes (Close Steps)**
Les étapes permettent de définir plusieurs étapes dans un modèle, exécutées soit séquentiellement, soit en parallèle.

**Exemple :**

```yaml
- name: hello-hello-hello
  steps:
  - - name: step1
      template: prepare-data
  - - name: step2a
      template: run-data-first-half
    - name: step2b
      template: run-data-second-half
```

## Template Types

Un template peut être des **containers**, **scripts**, **DAGs** (Directed-Acyclic Graphs) ou d’autres types selon la structure du workflow. Ils sont divisés en deux groupes :

* **Template Definitions** : définissent le travail à effectuer.
* **Template Invocators** : invoquent ou appellent d’autres templates et contrôlent l’exécution.

Sélectionnez « Expand » pour vous familiariser avec les différents types de templates.

---

### Template Definitions

**Container**
Un `Container` est le type de template le plus courant. Il représente une étape du workflow qui exécute un container. Convient pour des applications conteneurisées ou des scripts.

**Exemple :**

```yaml
- name: whalesay
  container:
    image: docker/whalesay
    command: [cowsay]
    args: ["hello world"]
```

---

**Resource**
Un `Resource` représente un template pour créer, modifier ou supprimer des ressources Kubernetes. Utile pour manipuler des objets Kubernetes dans le cluster sur lequel Workflows s’exécute.

**Exemple :**

```yaml
- name: k8s-owner-reference
  resource:
    action: create
    manifest: |
      apiVersion: v1
      kind: ConfigMap
      metadata:
        generateName: owned-eg-
      data:
        some: value
```

---

**Script**
Un `Script` est similaire à un `Container` mais permet de définir le script **inline**, sans référence à une image externe. Idéal pour des scripts simples ou des one-liners.

**Exemple :**

```yaml
- name: gen-random-int
  script:
    image: python:alpine3.6
    command: [python]
    source: |
      import random
      i = random.randint(1, 100)
      print(i)
```

---

**Suspend**
Un `Suspend` est un template qui suspend l’exécution, soit pour une durée donnée, soit jusqu’à reprise manuelle. La reprise peut se faire via la **CLI**, l’**API** ou l’**UI**.

**Exemple :**

```yaml
- name: delay
  suspend:
    duration: "20s"
```

---

### Template Invocators

**DAG (Directed-Acyclic Graph)**
Un `DAG` permet de définir les tâches comme un **graph de dépendances**. Utile pour les workflows avec des dépendances complexes et des exécutions conditionnelles.

**Exemple :**

```yaml
- name: diamond
  dag:
    tasks:
    - name: A
      template: echo
    - name: B
      dependencies: [A]
      template: echo
    - name: C
      dependencies: [A]
      template: echo
    - name: D
      dependencies: [B, C]
      template: echo
```

---

**Steps**
`Steps` permet de définir plusieurs étapes dans un template, à exécuter **séquentiellement** ou en **parallèle**.

**Exemple :**

```yaml
- name: hello-hello-hello
  steps:
  - - name: step1
      template: prepare-data
  - - name: step2a
      template: run-data-first-half
    - name: step2b
      template: run-data-second-half
```
